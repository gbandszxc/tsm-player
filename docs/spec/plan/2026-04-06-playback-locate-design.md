# 播放目录定位与浏览页布局调整设计

日期：2026-04-06

## 背景

当前 SMB 浏览器页的主体顺序为“文件浏览在上，播放控制在下”，与新的交互偏好不一致。播放页底部也缺少一个可直接把用户带回当前歌曲所在目录的入口。

现有实现虽然会持久化“上次播放”快照，但快照只包含队列、索引、进度和标题，不包含播放来源 SMB 连接信息。因此应用彻底退出后重新打开，无法可靠地把浏览页恢复到当前或上次播放歌曲所在目录，尤其是在存在多个 SMB 连接的情况下。

## 目标

1. SMB 浏览器页中，将“播放控制”区块调整到“文件浏览”区块上方。
2. 播放页在“返回文件页”按钮右侧增加黄色“定位”按钮。
3. 点击“定位”后，回到浏览页并显示当前播放歌曲所在目录。
4. 若当前无活跃播放，但存在上次播放记录，定位仍然可用。
5. 若目标歌曲属于其他 SMB 连接，先弹出确认框；用户确认后切换连接并跳转，取消则不执行任何动作。
6. 应用彻底退出后重新打开，上述定位能力仍然有效。

## 非目标

1. 不新增独立的“播放来源管理”页面。
2. 不重构当前整个 SMB 浏览数据结构。
3. 不改变“回到当前播放 / 继续上次播放”按钮的已有语义。

## 方案选择

采用“增强播放快照”的方案。

### 方案说明

扩展 `LastPlaybackStore.Snapshot`，除了现有的队列、索引、进度、标题外，再保存以下信息：

1. 当前播放项 `mediaId`
2. 当前播放项所在目录路径
3. 播放来源连接 ID
4. 播放来源 SMB 配置

这样无论定位发生在：

1. 播放页中的当前活跃播放
2. 浏览页中的“继续上次播放”
3. 应用完全退出后再次打开

都可以从统一的数据结构中恢复目标目录与目标连接，避免依赖进程内的 `PlaybackConfigStore`。

## 状态与数据流

### 播放快照写入

在播放页和播放服务保存快照时，同步写入：

1. 当前播放项 `mediaId`
2. 当前播放项目录路径
3. 当前来源连接 ID
4. 当前来源 SMB 配置

来源优先级：

1. 连接 ID 优先使用当前浏览页或配置存储中的 active connection
2. 配置优先使用 `PlaybackConfigStore.current()`
3. 若内存配置为空，则回退到 `SmbConfigStore.loadState().activeConfig`

### 定位请求解析

新增一个可复用的“播放定位目标解析”逻辑，输入为：

1. 当前 `MediaController` 的活跃播放项，或
2. `LastPlaybackStore.Snapshot`

输出为：

1. 目标目录路径
2. 目标连接 ID（若可确定）
3. 目标 SMB 配置
4. 是否与当前激活连接一致

### 浏览页跳转

浏览页 ViewModel 负责执行定位，而不是由播放页直接拼接浏览状态。定位过程：

1. 判断目标是否属于当前激活连接
2. 若属于当前连接：
   - 保存目标浏览路径
   - 更新当前路径
   - 加载目标目录
3. 若不属于当前连接：
   - UI 层弹出确认框
   - 用户确认后切换 active connection
   - 保存目标浏览路径
   - 加载目标目录
4. 用户取消时，保持当前页面状态不变

## 兼容性

老版本快照没有新增字段。读取时需要兼容：

1. 先读取新字段
2. 若新字段缺失，则尝试用 `currentIndex + queueMediaIds` 推导 `currentMediaId`
3. 再由 `currentMediaId` 反推父目录
4. 若仍无法解析，则视为“不可定位”

## 错误处理

1. 无活跃播放且无上次播放快照：按钮隐藏或提示“暂无可定位的播放记录”
2. 快照损坏或目标目录解析失败：提示“无法定位当前播放目录”
3. 目标连接已不存在，但快照中带有 SMB 配置：
   - 可以按目标配置直接切换为临时活动配置并加载目录
   - 不强制要求重新保存为命名连接
4. 目录加载失败沿用现有 SMB 错误映射

## 测试策略

1. 为快照新增字段与兼容回退逻辑补单测
2. 为“从播放信息解析目录与连接”的纯逻辑补单测
3. 为 `TvBrowserViewModel` 新增“定位到同连接目录”和“确认后切换到异连接目录”的单测
4. UI 层主要通过现有编译与行为链路验证，不引入新的 Android instrumentation test
